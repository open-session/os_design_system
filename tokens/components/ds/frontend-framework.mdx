# BOS Frontend Framework

> **In one paragraph.** BOS doesn't build UI from scratch. We pull primitives from Untitled UI Pro (and one day from ShadCN, Radix, Daisy — whichever the user picks), then apply our brand on top through two routes: **tokens** (one CSS file that retints everything automatically) and **wrappers** (a handful of files that override specific primitives where the brand needs them to *behave* differently). Six transform axes keep vendor source clean. Three wrapper shapes (A/B/C) handle case-by-case overrides. Six enforcement layers stop anyone from breaking the rules.

---

## The Three Folders

```
components/
├── base/      ← Vendor primitives (Untitled UI Pro, transformed)
├── custom/    ← Product compositions BOS authors from scratch
└── ds/        ← Design system formula spec — both mechanisms live here
```

The shape is enforced at error level: `.githooks/check-components-folders.sh` (pre-commit) plus `eslint-plugin-import/no-restricted-imports`. Adding a fourth top-level directory under `components/` is impossible.

| Folder | What lives here | Who edits | Source of truth |
|--------|----------------|----------|-----------------|
| **`base/`** | Vendor primitives — UUI Pro source files pulled via `bun run uui:add`. Sub-folders match UUI's taxonomy: `base/`, `application/`, `foundations/`, `shared-assets/`. | Codemod pipeline only — humans don't edit `base/` directly. | UUI Pro upstream (transformed). |
| **`custom/`** | Product compositions BOS authors. `pages/<route>/` (page-scoped) + `shared/<category>/` (cross-page, 9 categories). | Humans (Claude under enforcement). | BOS product code. |
| **`ds/`** | Design system formula spec. `brand.css` (Mechanism 1) + 5 wrapper files in `<primitive>/<name>.tsx` (Mechanism 2) + `defaults/`, `transforms/`, `code-quality/`, `motion/`, `shape/`, `accessibility/` MDX. | Humans. | BOS brand decisions. |

**Decision rule.** Vendor-derived → `base/`. Brand override → `ds/`. Product feature → `custom/`.

## The Two Mechanisms

```
┌──────────────────────────────────────────────────────────────────────────┐
│                     How brand reaches a primitive                        │
├─────────────────────────────────────┬────────────────────────────────────┤
│  Mechanism 1 — Tokens               │  Mechanism 2 — Structural Variants │
│                                     │                                    │
│  "What is the value of this thing?" │  "What does this variant DO?"      │
│                                     │                                    │
│  Lives in: components/ds/brand.css  │  Lives in: components/ds/<name>/   │
│                                     │                                    │
│  Flows automatically via CSS vars   │  Imported explicitly via barrel    │
│                                     │                                    │
│  Covers ~95% of brand decisions     │  Covers ~5% — the hard cases       │
│                                     │                                    │
│  Examples:                          │  Examples:                         │
│    • brand color → #FF6900          │    • Button primary = neutral with │
│    • motion-micro → 100ms           │      hover-reveal (not orange CTA) │
│    • radius-brand → 12px            │    • Tabs sizing scale restructure  │
│    • focus-ring brand → blue tint   │    • FeaturedIcon outline behavior │
└─────────────────────────────────────┴────────────────────────────────────┘
```

## Mechanism 1 — Tokens (the automatic 95%)

`components/ds/brand.css` is the single source of truth for **what every brand value equals**. It uses Tailwind's `@theme` to define CSS custom properties: `--bg-primary`, `--ring-brand`, `--motion-micro`, `--radius-brand`, and the full token vocabulary.

Vendor primitives in `components/base/` don't hardcode pixel values — they reference token names. UUI's Button says "use `bg-brand-solid`" — and `brand.css` decides what `bg-brand-solid` means today.

**Why this is mostly automatic:** Tailwind reads CSS variables at build time. Change a value in `brand.css`, and every Button / Input / Tab / FeaturedIcon across the app updates — no code edits, no imports. The vendor file doesn't know anything changed.

**The catch:** Mechanism 1 is automatic for VALUES within a stable vocabulary, not for the vocabulary itself. If the vendor renames or removes CSS custom properties, `brand.css` needs a coordinated update. This is a "Type 1" change (see below). The UUI v7→v8 upgrade reduced CSS variables from 1,093 to 657 (a 40% drop) — that was a real Type 1 event.

**What tokens cannot do:** They change values, not behaviors. There is no token called `--button-personality`. Token swaps don't restructure logic.

---

## Mechanism 2 — Wrappers (the explicit 5%)

Some brand decisions aren't values — they're behaviors. UUI's Button `primary` variant means "orange background, white text." BOS's `primary` means "neutral surface by default, brand color appears on hover." That is a different interaction pattern, not a color swap.

For these cases, BOS publishes its own version of the primitive at `components/ds/<primitive>/<primitive>.tsx`. The barrel at `components/base/index.ts` re-routes consumers to the BOS version. The vendor's pristine source stays at `components/base/<sub>/<primitive>/<file>.tsx` as a reference, but no application code routes through it.

**In Figma terms:** think of it as publishing your own variant of someone else's master component. You inherit their structure, override the parts that matter, and your team links to your variant — not the original.

Mechanism 2's wrappers come in **three shapes** (Shape A, B, C). Which one you use depends on how deep the override needs to go — see the dedicated section below.

## Axes vs Shapes — what's the difference?

This is the part people get confused about, so let's make it explicit.

You'll see two things named like this:

- **The six transform axes** (motion, focus, disabled, token syntax, devProps, vendor-primitive wrapper retention).
- **The three wrapper shapes** (Shape A, B, C).

They sound similar but they're different concepts.

| | **Axes (the six)** | **Shapes (the three)** |
| --- | --- | --- |
| What they describe | Things we **rewrite in vendor source code** automatically | Three different **kinds of wrapper file** we author by hand |
| Apply to | `components/base/` — vendor files we don't author | `components/ds/<primitive>/<name>.tsx` — wrapper files we DO author |
| When they run | Every `bun run uui:add` (codemod time) + write-time ESLint in `custom/` | Whenever we write or edit a wrapper file |
| Driven by | Mechanical rules — same edit applied uniformly everywhere | Per-primitive judgement — pick the lightest shape that covers the decision |

Put another way: **axes are about cleaning up someone else's code so it matches our conventions**; **shapes are about replacing someone else's component with our own version**. A primitive can be touched by axes (codemod transforms its motion classes, focus ring, etc.) without ever needing a wrapper — most primitives are like this (Input, Select, Badge, Avatar). Only when there's a *behavioral* brand decision do we add a wrapper.

---

## The Three Wrapper Shapes — when to use which

Three different *kinds* of wrapper file, in order of effort. Pick the lightest shape that covers the brand decision; escalate only when forced.

### Shape A — Behavioral re-export ("set different defaults")

**What it is.** A 5–10 line file. Imports the vendor primitive, calls it with new default props, re-exports the result. The vendor's actual rendering, classes, internal behavior — all unchanged. We just pre-fill the props.

**When to use it.** The brand decision is "we prefer different defaults" — no class changes, no rendering changes.

**BOS example today.** [Tooltip](./tooltips/tooltip.tsx). The vendor ships Tooltip with `delay=0, placement="top"`. BOS wants `delay=300, placement="top"` so tooltips feel less twitchy. Vendor's rendering, classes, a11y behavior — all identical. Just different default values for two props.

**Designer analogy.** *Same product, different settings out-of-the-box.* The vendor sells a toaster; we ship the same toaster but pre-configured to "bagel mode."

### Shape B — CVA override ("remap classes per variant")

**What it is.** A 30–100 line file. Re-implements the variant-mapping function (the `cva(...)` call that maps variant props like `size: "sm" | "md" | "lg"` to className strings). Keeps the vendor's JSX renderer — only the *classes* each variant produces are different.

**When to use it.** The vendor's variant arms exist and you want to keep them, but the class strings each variant produces should be different. The vendor's component structure is fine; only the visual output of variants needs to change.

**BOS example today.** None. We don't currently use Shape B — every brand decision so far has fit Shape A (Tooltip's defaults) or required Shape C (Button's calm-primary couldn't be expressed by remapping classes alone). Shape B sits in the middle; we'll likely use it the next time a primitive needs class-level deltas but no behavioral change.

**Designer analogy.** *Same product structure, repainted variants.* The vendor's Small/Medium/Large variants are in the right shape, but the paint colors don't match our brand — we repaint each variant without rebuilding the widget.

### Shape C — Full fork ("build your own from the same foundations")

**What it is.** A 200–500 line file. Re-implements the entire primitive against the same React Aria foundations the vendor uses. The vendor's source becomes a reference; at runtime, nothing reads from it — our wrapper is what renders.

**When to use it.** The brand decision crosses so many points that re-implementing is cleaner than continuing to override. Or the vendor's variant arms aren't exposed for override. Or the brand wants new variant arms / interaction patterns the vendor doesn't ship.

**BOS examples today.** Four wrappers:

- [Button](./buttons/button.tsx) — calm-primary (neutral by default, brand color appears on hover) + 5 secondary deltas. Different interaction pattern, not just classes.
- [Tabs](./tabs/tabs.tsx) — `ease-motion-out` retained for active-tab indicator + restructured sizing scale.
- [Modal](./modals/modal.tsx) — BOS Dialog surface treatment + slower modal motion.
- [FeaturedIcon](./featured-icon/featured-icon.tsx) — modern-theme + sm icon-stroke deltas.

**Designer analogy.** *Build your own version using the same internal parts.* The vendor's primitive is built on React Aria; when their pre-built primitive doesn't fit and the brand decision is structural rather than paintable, we rebuild from those same React Aria parts ourselves.

### The escalation rule

**Always try Shape A first.** Most brand decisions are just defaults. *Different delay = Shape A. Different placement = Shape A.* If you can express the decision as "the same primitive with different default props," you're done.

**Escalate to Shape B only when Shape A can't.** When you need different classes per existing variant arm.

**Escalate to Shape C only when Shape B can't.** When the brand wants new variant arms or a different interaction pattern. Shape C is the heaviest commitment — bigger file to maintain, each vendor update is a manual merge.

In BOS today: 4 Shape C, 1 Shape A, 0 Shape B. Shape B is the "missing middle" waiting for the right use case.

## The Six Transform Axes

Six **transform axes** describe the mechanical edits applied to vendor source on every `bun run uui:add` to align it with BOS conventions. Five of the six are also enforced as ESLint rules at write time in `components/custom/` so developer-authored code can't drift.

| # | Axis | What it does | Codemod fn | ESLint (write-time) | Doc |
|---|------|-------------|-----------|--------------------|-----|
| 1 | Motion tokens | `duration-200`/`ease-out` → BOS motion tokens | `applyMotionTokens` | `better-tailwindcss/no-restricted-classes` | [motion.mdx](./transforms/motion.mdx) |
| 2 | Focus ring | `ring-2 ring-offset-2` → `ring-1 shadow-focus-ring` | `applyFocusRing` | `better-tailwindcss/no-restricted-classes` | [focus-ring.mdx](./transforms/focus-ring.mdx) |
| 3 | Disabled state | Raw gray classes → `disabled:bg-bg-disabled_subtle` | `applyDisabledStates` | `better-tailwindcss/no-restricted-classes` | [disabled-state.mdx](./transforms/disabled-state.mdx) |
| 4 | Token syntax (Style 2) | Style 1 brackets → Style 2 mapped | `applyTokenSyntax` | `bos-local/prefer-mapped-tokens` (error) | [token-syntax.mdx](./transforms/token-syntax.mdx) |
| 5 | devProps | Inject `{...devProps('Name')}` on root DOM | `applyDevProps` | `bos-local/require-dev-props` (error) | [dev-props.mdx](./transforms/dev-props.mdx) |
| 6 | Vendor-primitive wrapper retention | Preserve cross-imports of `ds/` wrappers across re-pulls (parameterized over `WRAPPED_PRIMITIVES`) | `wrapperImportRetention` | codemod-only — operates on vendor source | [wrapper-import-retention.mdx](./transforms/wrapper-import-retention.mdx) |

All six axes live in `scripts/uui-apply-transformations.ts` and run inside `bun run uui:add`.

### How to add a new axis

The 6-axis design is parameterized — adding a 7th is mechanical:

1. Spec the rule in `components/ds/transforms/<axis-name>.mdx` (typed frontmatter: `type`, `axis`, `detection-command`, `enforcement`, `tags`).
2. Implement `applyMyAxis(source: string)` in `scripts/uui-apply-transformations.ts` — pure string transform, idempotent.
3. Register in `transformFile()` orchestrator (respect axis ordering).
4. Add a write-time enforcement rule (if applicable): configure `better-tailwindcss/no-restricted-classes` for class-name patterns, or write a custom rule at `eslint-rules/<rule-name>.js` and register in `eslint.config.mjs`.
5. Add unit tests at `tests/unit/scripts/fixtures/<axis-name>-*.ts`. The MDX docs reuse these fixtures as before/after examples.
6. Update [eslint-coverage.mdx](./code-quality/eslint-coverage.mdx) and this doc.

## The Figma Analogy (Whole-System View)

| Code concept | Figma analog |
|---|---|
| `components/ds/brand.css` | Color Styles + Effect Styles + Variables. One change cascades everywhere. |
| `components/base/` | Master components someone else made (the UUI Pro library). Has its own internal logic, variant defs, sizing scales. |
| `components/ds/<primitive>/` | Your "published variant" of a vendor master component. Inherits structure, overrides properties, gets used in production instead of the original. |
| `components/custom/` | Your designed pages and instances. Composes both DS primitives and one-off layouts. Where product UX actually lives. |
| `bun run uui:add <name>` | The "pull updates from the source library" action. Lets vendor changes flow into the master without breaking your published variants. |

**The dream:** change a brand value in `brand.css`, see every primitive update everywhere — without touching component code. **The reality:** that dream works perfectly for the ~95% of brand decisions that ARE just token values. For the structural ~5%, you publish a tiny explicit variant file and route consumers to it.

## How Import Resolution Actually Works

When application code writes `import { Button } from '@/components/base'`:

1. The barrel at `components/base/index.ts` intercepts the import.
2. For Button specifically, it re-exports from `@/components/ds/buttons/button` — NOT from `./base/buttons/button`.
3. The user's app renders the BOS Button with all brand decisions intact.
4. The vendor-pristine `base/base/buttons/button.tsx` exists as the "what UUI ships" baseline — only ever read by the codemod on re-pull.

**Production path:** consumers → barrel → `ds/` wrapper → render. The `base/` vendor file is a reference artifact, not a runtime dependency.

For primitives with **no structural deltas** (most of them), the barrel re-exports directly from `base/`. No wrapper, no detour. Tokens flow through automatically. Mechanism 1 alone is enough.

## Decision Tree — Which Mechanism Applies?

```
Is it a value change? (color, motion timing, radius, opacity, focus ring tint)
  → Mechanism 1. Edit components/ds/brand.css.
    Every primitive updates automatically. Done.

Is it a behavior change for one specific primitive?
  (variant arms redefined, sizing scale shifted, new conditional logic)
  → Mechanism 2. Create or edit components/ds/<primitive>/<primitive>.tsx.
    See components/ds/wrapper-template.mdx for the three wrapper shapes.

Is it a NEW primitive that doesn't exist anywhere yet?
  → Build it under components/custom/<page-or-shared-category>/.
    DS primitives are vendor-sourced; custom primitives are BOS-authored.

Is it a one-off layout for a single page?
  → components/custom/pages/<route>/. Compose DS primitives there.
```

## Three Types of Vendor Change

The two-mechanism model frames how you express brand. This section frames how the vendor changes underneath you. Every vendor update falls into one of three types.

### Type 1 — Token Vocabulary Changes (Rare, Consequential)

The vendor renames, removes, or adds CSS custom properties.

**Response:** Update `brand.css` to align with the new vocabulary. Mechanism 1 requires maintenance here.

**Risk if ignored:** `brand.css` defines orphaned tokens (vendor dropped them — bloats CSS) or fails to define new ones (vendor primitives reference undefined variables — silent visual bugs, missing colors, broken disabled states).

### Type 2 — API Additions (Common, Additive)

The vendor adds new props, sizes, variants, or whole primitives. UUI v8 added `lg` to Input and Select, `xs` to Button, `selectionIndicator` to Dropdown, and the entire multi-select / OTP input set.

**Response:** Adopt. Pull the updated `base/` files. No wrapper required unless there is a Type 3 reason to override.

**Risk if mistreated as Type 3:** Wrapping the new feature freezes BOS at the old vendor version — every subsequent vendor improvement to that primitive becomes a manual merge. Enough of these and the design system stops tracking the vendor at all.

### Type 3 — Behavior Overrides (Small, Deliberate)

You deliberately want the primitive to behave differently than the vendor ships. Examples: Button's calm-primary (Decision #2 — vendor ships orange CTA, BOS shows neutral with brand-on-hover), Tabs' restructured sizing scale, FeaturedIcon's outline behavior.

**Response:** Wrap at `components/ds/<primitive>/<primitive>.tsx`. This IS Mechanism 2. The wrapper isolates the brand decision from vendor evolution.

**Risk if leaky:** Letting the brand decision live inside `base/` directly. Vendor re-pull silently overwrites it, design system drifts back to vendor defaults, no one notices until a stakeholder asks why the primary button turned orange.

```
┌────────────────────┬───────────────────┬────────────────────────────────┐
│ Type 1             │ Type 2            │ Type 3                         │
│ Token vocabulary   │ API additions     │ Behavior overrides             │
├────────────────────┼───────────────────┼────────────────────────────────┤
│ Update brand.css   │ Adopt — no wrapper│ Wrap at ds/<primitive>/        │
│ Frequency: rare    │ Frequency: common │ Frequency: small but real      │
│ Risk: silent visual│ Risk: frozen at   │ Risk: vendor re-pull overwrites│
│ bugs               │ old vendor        │ brand decision silently        │
└────────────────────┴───────────────────┴────────────────────────────────┘
```

## Six Enforcement Layers

The architecture only works if no one bypasses it. BOS enforces the two mechanisms + six axes through six layers, each catching different failure modes at different times. The earlier the layer, the cheaper the catch.

| Layer | When it runs | What it catches | How |
|-------|-------------|----------------|-----|
| **1. Claude project config** ([.claude/CLAUDE.md](../.claude/CLAUDE.md)) | At generation time | Forbidden patterns (Sparkles, `border-2`, Style 1 brackets, raw grays). Three-folder rule. devProps requirement. UUI Transformation Rules (5 axes). | Loaded into every Claude Code session. Rules are imperatives — Claude won't generate violating code. |
| **2. Codemod (vendor-pull time)** | At `bun run uui:add` | All 6 transform axes against vendor source. | Runs automatically inside the pipeline. Cannot be skipped. |
| **3. TypeScript** | Write time + pre-commit | Type errors, missing imports, signature mismatches. | `bun run typecheck` (`tsc --noEmit`). Strict mode. |
| **4. ESLint (write-time, in `custom/`)** | Write time + pre-commit | 5 of 6 transform axes (Axis 6 codemod-only). Three-folder import boundaries. devProps. Story coverage. | `eslint.config.mjs` — `bos-local/*` plugin + `eslint-plugin-better-tailwindcss/no-restricted-classes`. See [eslint-coverage.mdx](./code-quality/eslint-coverage.mdx). |
| **5. Git pre-commit hook** | Commit time | Three-folder rule (no new top-level dirs). Secret scanning. Lint-staged subset. | `.githooks/pre-commit` + `.githooks/check-components-folders.sh`. |
| **6. CI (PR time)** | On every PR | Full lint, typecheck, build, test, Storybook build, Greptile review. | GitHub Actions + Vercel + Greptile. Branch protection on `main`. |

**Why Layer 1 matters.** Catching a `border-2` violation at Layer 1 (Claude doesn't generate it) costs nothing. Catching it at Layer 6 (Greptile flags it on a PR) costs a review round. Catching it at Layer 7 (a designer notices visual drift) costs a bug fix + retro. Every catch we move earlier compounds.

## Current State (PRD 020 Shipped, 2026-05-09)

**Mechanism 1:** Fully working but not free. Phase C confirmed that token alignment is earned, not automatic. Token maintenance is required on every major upgrade.

**Mechanism 2:** 5 wrappers retained (not 23):

| Wrapper | Shape | Brand Decision |
|---------|-------|---------------|
| `ds/buttons/button.tsx` | C (full fork) | Decision #2 calm-primary + 5 secondary deltas |
| `ds/tabs/tabs.tsx` | C (full fork) | `ease-motion-out` retained; deletable if codemod axis 1b lands |
| `ds/featured-icon/featured-icon.tsx` | C (full fork) | modern-theme + sm icon-stroke deltas |
| `ds/modals/modal.tsx` | C (full fork) | BOS Dialog surface + slower modal motion |
| `ds/tooltips/tooltip.tsx` | A (behavioral) | Default delay=300, placement=top (PRD 020 Wave 2) |

The wrapper-deletion test held end-to-end: every wave applied the three-way diff (current / vendor-pristine / wrapper) and dropped any wrapper without surviving Type 3 deltas. The Phase B regex sweep predicted 31 wrappers; the post-sweep refinement said ~26; the actual count was 5 (4 structural + 1 behavioral).

**The wrapper-deletion test:** When evaluating a wrapper candidate, ask: "Are there still ≥1 Type 3 deltas remaining?" If yes, wrapper survives. If no, delete the wrapper, re-route the barrel to `base/` directly. Default: lean into UUI and drop wrapper candidates unless product use is currently validated.

## Vendor Agnosticism

| Piece | Vendor-agnostic? | Coupling |
|-------|-----------------|---------|
| `ds/brand.css` (Mechanism 1) | Fully — any Tailwind-based vendor reads tokens automatically | None |
| `ds/<primitive>/` wrappers (Mechanism 2) | Conceptually (three shapes work for any vendor) | Implementation imports vendor-specific symbols; a swap means rewriting wrapper bodies |
| `bun run uui:add` pipeline | UUI-specific today | Hardcoded to `npx untitledui@latest`; needs vendor adapter layer for PRD 018 |

**For a vendor swap (PRD 018):** Mechanism 1 transfers for free. Mechanism 2 wrappers need re-authoring against the new vendor's primitives. See [Swap Guide](./swap-guide.mdx).

## Product Vision — From Internal Architecture to User Feature

This codebase is a working proof of three ideas we want to expose to BOS users.

### Vision 1 — "Switch your UI vendor"

Today BOS as a product is hard-wired to UUI Pro. The architecture lets us swap to ShadCN / Radix / Chakra / Daisy internally with a focused PRD (018 territory). The product vision is to expose that swap as a **user-facing flow**: a tenant chooses a vendor at sign-up (or switches mid-flight), the `WRAPPED_PRIMITIVES` config + axis regex tables become per-vendor mappings stored on the backend, the same `brand.css` token set drives the chosen vendor's primitives, wrapper bodies are re-generated for the new vendor automatically. The user's BOS instance renders against the new vendor with their existing brand intact.

Blocker today: `bun run uui:add` is UUI-CLI-specific. The vendor adapter layer is the work item. Once it lands, "vendor swap" becomes a setting, not a campaign.

### Vision 2 — "Manage a design system inside the product"

Today, designers and developers manage the BOS design system in this repo (file edits + Storybook + PRs). The product vision is to expose **the same surface inside BOS itself** — like Storybook, but for the user's own brand:

- **Tokens UI.** A panel that edits `brand.css` token values live, with previews on every primitive. Drives Mechanism 1 visually. (Mostly already feasible — a managed-agents skill could handle the brand.css mutations under guard rails.)
- **Wrapper authoring UI.** A panel that takes a primitive + a brand decision ("our primary button is calm with hover-reveal") and generates a Shape A/B/C wrapper file. Drives Mechanism 2 visually. (Harder — needs the codemod + AST tooling exposed as a service.)
- **Transform-axis recommendations.** When a user pulls a vendor update, the product surfaces "the vendor changed these tokens; here's how to update your brand" — using the Type 1/2/3 framework as the diagnostic.
- **Theme + token agent.** Anthropic's Managed Agents take a high-level brand brief ("warm, confident, slightly playful") and produce a candidate `brand.css` set the user can iterate on. Cheaper than designing every token from scratch.

Blocker today: codemod and ESLint enforcement are CLI-bound, not service-callable. Wrapper authoring is human-only. The product surface (UI panels) doesn't exist yet. But the underlying architecture is shaped exactly for this.

### Vision 3 — Marketplace (`@opensession/brand-os-spec`)

Once Vision 1 + 2 land, the same spec can ship to other teams as `@opensession/brand-os-spec` — a portable formula spec any team can use against any vendor. BOS becomes the *infrastructure for design-system management*, not just one company's design system.

Blocker today: a second adopter showing up. Premature now; tracked.

## Known Gaps

- **No Type 1 awareness in the pipeline.** `bun run uui:add` doesn't check whether a vendor pull references undefined tokens or leaves orphaned ones.
- **No vendor major-version mode.** The pipeline is one-primitive-at-a-time. Major versions are coordinated multi-primitive events.
- **No `brand.css` ↔ vendor diff tooling.** Type 1 changes are caught only by manual inspection or downstream visual bugs.
- **Codemod incomplete on multi-export files.** `scripts/uui-apply-transformations.ts` only injects `devProps` on the first exported component.
- **Codemod missing `ease-linear` rule.** Both Button and Tabs have `ease-linear` upstream that BOS prefers as `ease-motion-out`.
